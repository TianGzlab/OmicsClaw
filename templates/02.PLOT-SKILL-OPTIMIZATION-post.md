# OmicsClaw Unified Skill Optimization Template

本文档用于总结当前已经在 `skills/spatial` 多个 skill 中验证过的统一优化模板，目标是为后续：

- `skills/spatial/*`
- `skills/singlecell/*`
- `skills/genomics/*`
- `skills/proteomics/*`
- `skills/metabolomics/*`
- `skills/bulkrna/*`

提供一套可复用、可扩展、可验证的迁移规范。

当前已经按这一思路迁移或验证过的代表性 skill 包括：

- `spatial-annotate`
- `spatial-genes`
- `spatial-domains`
- `spatial-integrate`
- `spatial-cnv`
- `spatial-communication`
- `spatial-condition`
- `spatial-de`
- `spatial-deconv`

---

## 1. 模板的核心目标

这套模板不是单纯“把代码整理一下”，而是同时解决 5 个问题：

1. **用户体验统一**
   - 不同 skill 的输出层次一致。
   - 用户知道先看哪里，再看哪里，后续怎么二次美化。

2. **方法扩展友好**
   - 新方法加入时，不需要重写整套报告和绘图逻辑。
   - 方法差异尽量收敛到 `param_hints`、结果持久化和局部 renderer。

3. **结果可持久化**
   - 绘图和报告不能依赖运行时临时变量。
   - 关键结果必须写回 `adata.obs` / `adata.obsm` / `adata.uns` 或标准表格。

4. **可复用可外接**
   - Python 标准画廊是 canonical layer。
   - R 可视化层是 optional styling layer，消费 `figure_data/`，不重算科学结果。

5. **后续可测试可审计**
   - 每个 skill 都能有稳定的文件契约、manifest 契约、测试契约、文档契约。

## 2. 统一设计原则

### 2.1 Python 图是标准结果层

Python 层负责：

- 生成 OmicsClaw 默认分析叙事
- 产出标准 `figures/`
- 产出可供下游消费的 `figure_data/`
- 与 `report.md` / `result.json` / `processed.h5ad` 保持一致

它是默认应该被用户首先看到和解释的一层。

### 2.2 R 图是专门的可视化增强层

R 层负责：

- 读取 `figure_data/*.csv`
- 做版式、主题、标注、组合图、投稿风格排版
- 输出到 `figures/custom/`

R 层不应负责：

- 重新跑差异分析
- 重新跑注释/去卷积/通讯推断
- 在绘图脚本里隐藏式重算核心统计结果

### 2.3 画图逻辑必须声明式

不能继续把 `generate_figures()` 写成一大段 `if/else + savefig`。

应改为：

- `PlotSpec`
- `VisualizationRecipe`
- `render_plot_specs(...)`

即：

1. skill 先声明“应该产出哪些图”
2. 再由 renderer 映射到具体绘图函数
3. 再自动写入 `figures/manifest.json`

### 2.4 画图依赖的状态必须先标准化

不能让 figure 直接吃“前面函数临时返回的局部变量”。

正确做法：

1. 方法跑完后，先把核心结果写入持久结构
   - `adata.obs`
   - `adata.obsm`
   - `adata.uns`
   - `tables/*.csv`
2. 然后由 gallery context 从这些稳定结构里构造绘图状态

这正是之前 `spatial-annotate` / `spatial-deconv` 中发现并修复的关键问题。

### 2.5 优先复用 `skills/<domain>/_lib/viz`

`_lib/viz` 应作为共享 renderer 原语层，而不是被闲置。

推荐优先级：

1. **先复用已有 `_lib/viz` 函数**
   - 如 `plot_features`
   - `plot_expression`
   - `plot_deconvolution`
   - `plot_communication`
   - 其他域内已有绘图原语
2. **skill 内加轻量包装 renderer**
   - 例如柱状图、火山图、summary 图、histogram
3. **只有当多个 skill 都需要时，再上提到 `_lib/viz`**

原则是：

- 通用原语进 `_lib/viz`
- skill-specific 叙事留在 skill 自己

## 3. 统一三层架构

统一建议分三层：

### 3.1 第 1 层：底层分析与方法封装层

负责：

- CLI 参数解析
- 方法选择
- 参数校验
- 运行后端分析
- 将核心结果写回标准数据结构

这一层的重点不是画图，而是保证：

- 方法特异参数正确暴露
- 结果被稳定存储
- wrapper 行为与文档一致

### 3.2 第 2 层：Python 标准画廊层

负责：

- 根据稳定结果构建 `gallery_context`
- 声明 `VisualizationRecipe`
- 渲染 `figures/`
- 导出 `figure_data/`
- 写 `figures/manifest.json`
- 写 `figure_data/manifest.json`

这一层是 OmicsClaw 默认 narrative layer。

### 3.3 第 3 层：R 可视化增强层

负责：

- 提供 `r_visualization/README.md`
- 提供一个 `*_publication_template.R`
- 自动生成 `reproducibility/r_visualization.sh`

这一层是用户可选层，不应该反向污染分析主链。

## 4. 每个 skill 的推荐代码结构

推荐在 skill 脚本里按下列顺序组织：

### 4.1 参数与运行配置

- `_build_parser()`
- `_validate_args(...)`
- `_collect_run_configuration(...)`
- 可选：
  - `_resolve_legacy_*`
  - `_append_cli_flag(...)`

目标：

- 参数暴露清晰
- CLI 与 `SKILL.md` 的 `allowed_extra_flags` / `param_hints` 一致
- 只记录“当前方法真正相关”的参数

### 4.2 Gallery helpers

这是迁移的核心区域，建议统一具备以下函数族：

- `_prepare_*_plot_state(...)`
  - 修正坐标别名
  - 补齐分类列 dtype
  - 恢复绘图所需 obs/obsm 状态

- `_ensure_umap_for_gallery(...)`
  - 标准画廊如果需要 embedding 视图，可以 best-effort 计算 UMAP

- `_build_*_summary_table(...)`
  - 运行摘要表
  - top hits 表
  - burden / diversity / uncertainty 表

- `_annotate_*_metrics_to_obs(...)`
  - 把供绘图使用的派生指标写回 `adata.obs`
  - 如：
    - dominant label
    - confidence
    - entropy
    - assignment margin
    - burden score
    - hub score
    - effect size count

- `_prepare_*_gallery_context(...)`
  - 统一构造 skill 的 gallery context
  - 这是标准画廊的唯一数据入口

- `_build_*_visualization_recipe(...)`
  - 返回 `VisualizationRecipe`

- `_render_*`
  - renderer 映射函数

- `*_GALLERY_RENDERERS`
  - renderer 注册表

- `_write_figure_data_manifest(...)`
- `_export_figure_data(...)`
- `generate_figures(...)`

### 4.3 Report / export helpers

建议统一具备：

- `export_tables(...)`
- `_write_r_visualization_helper(...)`
- `write_report(...)`

### 4.4 主流程

`main()` 统一做：

1. parse args
2. load data
3. collect method config
4. run analysis
5. persist core results to `adata`
6. build `gallery_context`
7. `generate_figures(...)`
8. `export_tables(...)`
9. `write_report(...)`
10. save `processed.h5ad`

## 5. 标准画廊的角色定义

当前统一建议每个 skill 的 recipe 至少围绕以下 4 类角色组织：

### 5.1 `overview`

主结果图。

用于回答：

- 这个 skill 的核心结论是什么？
- 用户第一眼应该看哪张图？

例子：

- annotation 的 cell type spatial map
- genes 的 top SVG spatial map
- domains 的 tissue domains map
- deconv 的 spatial proportions / dominant cell type
- communication 的 LR dotplot / heatmap

### 5.2 `diagnostic`

方法诊断图。

用于回答：

- 这个方法在当前数据上表现稳定吗？
- 是否有明显技术问题、混杂、模糊区、不收敛迹象？

例子：

- diversity / entropy map
- batch mixing before/after
- burden on tissue / UMAP
- uncertainty / confidence map
- spatial score diagnostic

### 5.3 `supporting`

辅助解释图。

用于回答：

- 主结论背后的结构性支持是什么？
- 哪些 summary 值能帮助快速解释结果？

例子：

- top-hit barplot
- cell type distribution
- group/domain burden summary
- mean proportions
- source-target summary

### 5.4 `uncertainty`

不确定性/稳定性图。

用于回答：

- 哪些区域/对象最不稳定？
- 结果的不确定性怎么解释？

例子：

- p-value histogram
- assignment margin histogram
- skipped contrasts / skipped sample groups
- sample support / replicate support
- entropy / ambiguity distribution

## 6. PlotSpec / VisualizationRecipe 的统一要求

每个 `PlotSpec` 建议至少包含：

- `plot_id`
- `role`
- `renderer`
- `filename`
- `title`
- `description`
- `params`
- `required_obs`
- `required_obsm`
- `required_uns`

推荐规则：

### 6.1 `plot_id`

- 全 skill 唯一
- 语义明确
- 用 `{skill_short}_{meaning}` 风格

例子：

- `annotation_spatial_overview`
- `deconv_assignment_margin_histogram`
- `condition_volcano_overview`

### 6.2 `filename`

应该稳定、直观、可预测。

避免：

- `plot1.png`
- `output.png`
- `tmp_figure.png`

推荐：

- `cell_type_spatial.png`
- `de_volcano.png`
- `assignment_margin_distribution.png`

### 6.3 `required_*`

这是模板里非常重要的一点。

必须显式声明每张图依赖什么，否则：

- 很难知道为什么图没出
- 很难在 manifest 中表达 skip reason
- 很难做跨 skill 复用

## 7. Gallery context 的标准职责

`gallery_context` 是统一模板里的关键抽象。

它的职责是：

1. **屏蔽分析层与画图层之间的耦合**
2. **把派生指标集中计算一次**
3. **为 report / result.json / figure_data 提供一致数据来源**

### 7.1 `gallery_context` 里应该放什么

建议放：

- 绘图必需的标准键
  - `spatial_key`
  - `prop_key`
  - `embedding_key`
  - `groupby`
  - `domain_col`
  - `confidence_col`
- 派生 DataFrame
  - `top_df`
  - `summary_df`
  - `metrics_df`
  - `counts_df`
  - `point_df`
- 回写到 `adata.obs` 后的新列名
  - `dominant_label_col`
  - `assignment_margin_col`
  - `entropy_col`
  - `effect_col`

### 7.2 `gallery_context` 不应该放什么

- 不稳定的本地临时变量名
- 很大的原始矩阵副本，除非确实需要
- 只能在当前函数栈里解释清楚的状态

## 8. 结果持久化规范

统一模板要求：**凡是后续要画、要汇报、要导出、要测试的结果，都必须先持久化。**

### 8.1 优先写入位置

#### `adata.obs`

适合逐 observation 的标量或标签：

- cell type
- domain
- confidence
- assignment margin
- entropy
- burden
- sender / receiver / hub score

#### `adata.obsm`

适合 observation x feature 的矩阵：

- deconvolution proportions
- probability matrices
- embeddings
- communication score matrices

#### `adata.uns`

适合：

- cell type names
- metadata
- method config
- top table
- summary dict

### 8.2 命名规范

推荐使用：

- `{analysis}_{method}`
- `{analysis}_{method}_metadata`
- `{analysis}_{method}_cell_types`
- `{skill}_{meaning}`

例子：

- `deconvolution_cell2location`
- `deconvolution_cell2location_metadata`
- `deconv_cell2location_assignment_margin`
- `condition_cluster_n_effect_hits`

### 8.3 严禁的反模式

- 图直接依赖某个前序函数返回值，但这个值没写到 `adata`
- report 依赖局部变量，而 rerun 时拿不到
- figure_data 导出和 report 用的是两套不同的数据来源

## 9. `figure_data/` 标准契约

`figure_data/` 是 Python canonical layer 与 R customization layer 之间的桥。

### 9.1 必备文件

每个 skill 迁移后，建议至少有：

- `figure_data/manifest.json`
- 若干 `figure_data/*.csv`

### 9.2 `manifest.json` 推荐字段

建议至少包含：

- `skill`
- `version`
- `method`
- `recipe_id`
- `gallery_roles`
- `available_files`
- `gallery_outputs`

可选：

- `groupby`
- `reference`
- `cell_type_key`
- `domain_column`
- `prop_key`
- `embedding_key`

### 9.3 `available_files`

这是最关键的桥接字段。

它应告诉下游：

- 哪个逻辑文件对应哪个 CSV
- 哪些文件可选
- 哪些文件在当前 run 中没生成

推荐写法：

```json
{
  "available_files": {
    "proportions": "proportions.csv",
    "deconv_spot_metrics": "deconv_spot_metrics.csv",
    "deconv_spatial_points": "deconv_spatial_points.csv",
    "deconv_umap_points": "deconv_umap_points.csv",
    "card_refined_proportions": null
  }
}
```

### 9.4 `gallery_outputs`

必须记录每张图的：

- `plot_id`
- `role`
- `filename`
- `status`

这样：

- 前端或下游工具可直接知道哪些图成功生成
- 用户可解释 skip / failed 的原因

---

## 10. `figures/` 标准契约

每个 skill 统一使用：

- `figures/manifest.json`

由 `render_plot_specs(...)` 自动写出。

它的职责是描述：

- 当前 recipe 是什么
- 每张图是否渲染成功
- 如果没成功，是 skip 还是 failed

推荐把它视为：

- Python gallery 的 index
- 不是报告内容本身，但能为上层 UI / agent / tool 提供统一查询入口

## 11. R 可视化层的统一契约

每个迁移后的 skill 都建议补：

```text
skills/<domain>/<skill>/r_visualization/
├── README.md
└── <template>.R
```

同时在输出目录下自动写：

```text
reproducibility/r_visualization.sh
```

### 11.1 `README.md` 应说明

- 这个 R 层是 optional
- Python gallery 才是 canonical output
- R 层吃哪些 `figure_data/*.csv`
- 模板产出到哪里
- 用户如何运行

### 11.2 R 模板应具备的性质

- 输入明确
- 最少依赖
- 不重算科学分析
- 只做美化和组合
- 输出到 `figures/custom/`

### 11.3 R 模板推荐最小范式

至少做一到两张最能代表这个 skill 的图：

- 一个主图
- 一个 supporting / uncertainty 图

例如：

- deconv：dominant map + mean proportions + assignment margin
- de：volcano + top hits
- domains：domain map + domain counts
- genes：top SVG map + score summary

## 12. `report.md` 的统一写法

`report.md` 不应只是堆参数。

建议统一包含：

1. `Summary`
2. 方法关键结果
3. `Parameters`
4. `Effective Method Parameters`
5. `Interpretation Notes`
6. `Visualization Outputs`

### 12.1 `Visualization Outputs` 段建议固定

```md
## Visualization Outputs

- `figures/manifest.json`: Standard Python gallery manifest
- `figure_data/`: Figure-ready CSV exports for downstream customization
- `reproducibility/r_visualization.sh`: Optional R visualization entrypoint
```

### 12.2 `result.json` 建议额外记录

在 `data.visualization` 中记录关键列名或对象名：

- `recipe_id`
- `prop_key`
- `dominant_label_column`
- `confidence_column`
- `assignment_margin_column`
- `entropy_column`

这样后续 agent 或前端可以不读整份报告，也能知道结果对象在哪。

## 13. `SKILL.md` 的统一升级点

每个 skill 在迁移时，不只要改 Python 脚本，还要同步升级 `SKILL.md`。

### 13.1 Frontmatter

重点检查：

- `version`
- `allowed_extra_flags`
- `param_hints`
- `legacy_aliases`
- `saves_h5ad`
- `requires_preprocessed`

### 13.2 `allowed_extra_flags`

规则：

- 只放当前 wrapper 真实支持、而且希望暴露给 orchestrator / bot 的 flag
- 不放历史遗留但实际上不应继续推荐的 flag
- 不放内部调试 flag

### 13.3 `param_hints`

这部分非常重要，是方法特异优化的核心。

每个方法都建议显式写：

- `priority`
- `params`
- `defaults`
- `requires`
- `tips`

要求：

- 参数必须与当前代码真实一致
- 默认值必须与当前 CLI/runtime 行为一致
- tips 要强调“用户第一步真正该调什么”

### 13.4 正文结构建议统一增加

- `Core Capabilities`
- `Workflow`
- `Visualization Contract`
- `Output Structure`
- `Safety`

其中 `Visualization Contract` 是当前模板的关键新增段。

## 14. knowledge base 的统一升级点

每次迁移建议同步检查：

- `knowledge_base/skill-guides/...`
- `knowledge_base/knowhows/...`

### 14.1 skill guide 应补什么

建议补：

- 如何选方法
- 方法间不要混淆的统计语义
- Python gallery 与 R layer 的职责分层
- 当前 skill 的 figure_data 关键文件说明

### 14.2 guardrails 应补什么

建议补：

- 运行前必须检查的输入前提
- 方法间不能混说的统计假设
- Python / R 两层可视化边界
- uncertainty 指标如何正确解释

## 15. 测试契约

统一模板迁移后，每个 skill 至少应补以下测试类型。

### 15.1 CLI / config 测试

验证：

- `_collect_run_configuration(...)` 只保留当前方法相关参数
- `commands.sh` 不包含无关方法的 flag

### 15.2 Gallery contract 测试

用 synthetic data 验证：

- `figures/manifest.json` 存在
- `figure_data/manifest.json` 存在
- 至少存在一组关键 `figure_data/*.csv`
- manifest 中包含 4 种 role

### 15.3 持久化测试

验证：

- gallery 派生列已写回 `adata.obs`
- 核心矩阵已写入 `adata.obsm`
- 元数据已写入 `adata.uns`

### 15.4 report / result 测试

验证：

- `report.md` 包含 `Visualization Outputs`
- `result.json` 包含 `data.visualization`
- `r_visualization.sh` 存在

### 15.5 demo 不可行时的策略

不是每个 skill 都适合真实 demo。

如果 demo 依赖外部 reference / 重资源 / 特定环境：

- 可以保留“demo 失败但信息明确”的测试
- 再用 synthetic unit tests 覆盖 gallery / export / report 契约

这比强行伪造一个误导性的 demo 更好。

## 16. 迁移时的推荐执行顺序

建议对每个 skill 按以下顺序迁移：

1. **先看当前脚本**
   - 核心结果现在存在哪
   - 哪些图依赖临时变量
   - 哪些参数是方法特异的

2. **补持久化**
   - 把后续需要解释/绘图的关键结果写回 `adata`

3. **搭 gallery context**
   - 统一派生 summary / metrics / obs 列名

4. **改造 `generate_figures()`**
   - 变为 recipe-driven

5. **导出 `figure_data/`**
   - 明确桥接 CSV

6. **补 R 层**
   - `r_visualization/README.md`
   - `*_publication_template.R`
   - `r_visualization.sh`

7. **升级 `report.md` / `result.json`**

8. **升级 `SKILL.md` / knowledge base**

9. **补测试**

10. **跑验证**
    - `py_compile`
    - `pytest`
    - 如条件允许，再跑一次 `r_visualization.sh`

## 17. 什么时候该复用 `_lib/viz`，什么时候不该

### 17.1 应优先复用 `_lib/viz` 的情况

- 空间 feature map
- embedding feature map
- dotplot / heatmap / expression map
- deconvolution 标准视图
- communication 标准视图

### 17.2 应在 skill 内局部实现 renderer 的情况

- 这个图只服务于当前 skill 的分析叙事
- 数据结构明显是 skill 特有的 summary table
- 不值得抽成通用原语

例如：

- top hits barplot
- skipped contrast summary
- assignment margin histogram
- per-cluster burden chart

### 17.3 应上提到 `_lib/viz` 的情况

如果一个 renderer 满足以下条件，可以考虑上提：

1. 至少两个以上 skill 都需要
2. 输入结构相对稳定
3. 不依赖当前 skill 私有语义

## 18. 跨组学迁移时如何泛化

这套模板不仅适用于 spatial。

### 18.1 不变的部分

以下部分在所有 omics 域都可以保留：

- 方法特异参数前置化
- recipe-driven Python gallery
- `figure_data/` 桥接层
- R customization layer
- `report.md` / `result.json` / `manifest` 契约
- synthetic contract tests

### 18.2 需要按域替换的部分

#### spatial

常见 basis：

- `spatial`
- `umap`
- `pca`

常见 point export：

- `*_spatial_points.csv`
- `*_umap_points.csv`

#### singlecell

可替换为：

- `cell-level embeddings`
- pseudotime projections
- cluster summaries
- gene program scores

#### genomics

可替换为：

- variant tables
- CNV segment summaries
- sample-level burden tables
- Manhattan / volcano / heatmap contracts

#### proteomics / metabolomics / bulkrna

可替换为：

- sample-level abundance tables
- pathway score tables
- volcano points
- batch summary tables
- feature matrices for publication plotting

### 18.3 非空间域的核心适配规则

即使没有 `spatial` 坐标，也保留同样思想：

- `figures/` 是 canonical layer
- `figure_data/` 是 bridge layer
- R 是 style layer
- `PlotSpec.role` 仍然保留四类：
  - `overview`
  - `diagnostic`
  - `supporting`
  - `uncertainty`

## 20. 统一反模式清单

以下做法应尽量避免：

1. **在 `generate_figures()` 里直接写大量 if/else**
2. **图依赖局部临时结果，没有持久化**
3. **R 层偷偷重算科学结果**
4. **`allowed_extra_flags` 把所有 CLI 都暴露出去**
5. **`param_hints` 和真实代码不一致**
6. **`result.json` 塞入巨大的 DataFrame**
7. **manifest 缺失，导致上层无法统一消费**
8. **测试只测 demo，不测契约**
9. **输出结构每个 skill 都不一样**
10. **同一指标在 report / figure_data / result.json 中用不同名字**

- 用户先看 Python 标准画廊即可理解主要结果
- 用户如果想进一步美化，可直接用 `figure_data/` + R 模板继续加工

## 23. 后续推荐实践

后面继续迁移时，建议以这份文档为标准，不再每个 skill 重新发明一套结构。

更具体地说：

1. **先统一架构，再讨论局部图形风格**
2. **先统一输出契约，再讨论方法扩展**
3. **先统一测试契约，再讨论 demo 丰富度**
4. **先统一 Python canonical layer，再逐步丰富 R publication layer**

这样做的收益是：

- 用户体验一致
- agent 更容易自动选择和解释 skill
- 后续新方法接入成本明显下降
- 跨域迁移时可以保留大部分工程骨架

建议把本文档作为后续迁移的总模板使用：

这份文档的定位不是“最终说明书”，而是 OmicsClaw 统一技能优化的工程基线。
